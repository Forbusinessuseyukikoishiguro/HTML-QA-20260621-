# 🐰 HTMLツール商用QA再現プロンプト集

> v7.0「うさうさ学習テキスト・ジェネレーター」のQAログ／テストレポート／作業ログから逆算した、単発HTMLツール開発に再利用できるプロンプト集。
> 全プロンプトはClaude CodeやCursor等のAIコーディングエージェントにそのまま投げられる形式。各プロンプトの直後に「なぜそうするか」を添える。

---

## 0. この手法の骨格（先に結論）

3層構成＋2回連続合格を出荷条件にする。

| 層 | 検証対象 | 拾える不具合 | 拾えない不具合 |
|---|---|---|---|
| T1 構文 | `node --check` | SyntaxError・閉じタグ漏れ | ロジック誤り全般 |
| T2 機能/UI/セキュリティ | jsdom上での単体・結合テスト | 状態遷移ミス・XSS・ボタンの空打ち・CSVインジェクション | 実バイナリ生成・実ブラウザ専用API |
| T3 実体検証 | 実際にファイルを書き出す | pptx 0バイト・壊れたzip・ライブラリ未同梱 | UI操作系の不具合 |

**なぜ3層か**：jsdomは「DOM上の見た目と状態」は正確に検証できるが、`Blob`から実際に有効なpptx/zipが生成されたかまでは保証しない。逆に実ファイル生成テストだけだとUIの結合不具合（編集→プレビュー反映漏れ等）を拾えない。両方ないと「動くが壊れている」「正しいが触れない」のどちらかを見逃す。

---

## 1. プロンプト①：QA憲法（全体方針を最初に固定する）

```text
あなたはこのHTMLツールの商用QA担当です。以下を絶対方針として守ってください。

1. 嘘の合格報告はしない。失敗したテストは失敗のまま記録する。
2. テストは「最終確認1回目」「最終2回目」の連続2回実施し、両方合格して初めて出荷可とする。
3. テストが失敗した場合、原因が「アプリ本体の不具合」なのか「テストコード（治具）側の不備」なのかを
   必ず切り分けてから報告する。切り分けずに「直しました」とだけ言うのは禁止。
4. 検証できなかった項目（実ブラウザでしか確認できない非同期処理など）は
   「未実証」として正直に明記し、検証済みであるかのように書かない。
5. 各テスト結果は ✅/❌ と日本語の短い説明をセットで出力する。
```

**なぜ**：v7.0のQAログ・レポートに一貫する「嘘つかず・2回」という方針そのもの。これを最初に固定しないと、AIエージェントは（褒められたいバイアスで）都合よく「合格」を盛りがちになる。憲法として先に宣言することで、後続の全プロンプトの判断基準になる。

## 2. プロンプト②：T1 構文チェック

```text
T1: 構文チェックを実施してください。

- 対象: usausa_xxx_vN.html 内に埋め込まれた <script> の中身をすべて抽出
- コマンド: node --check で構文エラーの有無を確認
- 出力形式:
  [T1] node --check
    ✅ T1   もしくは
    ❌ T1: <エラー内容>
```

**なぜ**：HTMLファイル全体は`node --check`の対象にならないため、scriptタグの中身だけを切り出して検証する必要がある。これを最初のゲートにするのは、文法エラーがある状態でT2(jsdom)を回しても「テストが落ちたのか構文が壊れているのか」の切り分けが無駄に複雑になるため。最も安く・最も早く落とせる層を先頭に置く。

## 3. プロンプト③：T2 jsdom統合テストスイート（カテゴリA〜L）

ここがQAの本体。**カテゴリごとに分割して書かせる**のがコツ（一括で書かせると抜け漏れが出やすい）。

```text
T2: jsdomを使い、以下のカテゴリ全てについてテストを作成・実行してください。
1カテゴリ＝1関数として独立させ、最後に「合計N passed / M failed」を集計してください。
各テストは「何を検証しているか」がテスト名から分かるようにしてください。
```

そのうえで、以下の表のプロンプトを順に投げる（または1メッセージにまとめて全カテゴリ列挙）。

| カテゴリ | 検証プロンプト（要約） | なぜこのカテゴリが必要か |
|---|---|---|
| A) 同梱ライブラリ | `PptxGenJS`と`JSZip`がHTML内に正しく埋め込まれグローバルから参照できるか検証 | オフライン単体HTMLが前提のため、CDN依存ゼロを物理的に保証する必要がある |
| B) 生成→目次＋見開き | 生成後に目次へ項目が並ぶか、見開き表示(grid)になるか、placeholderが消えるか | UI再設計の主目的（タブめくりUI廃止）が実際に達成されているかの根拠 |
| C) ページ1レイアウト | 初期セクションがoverviewか、左に企画全体・目的が出るか、右にプレビューが出るか | 「左編集・右プレビュー」という設計原則の起点ページが壊れていないか |
| D) セクション編集 | 左がtextareaか、右にh1/h2レンダリングされるか、編集がSTATEに反映され続けるか | 双方向バインディングはバグの温床。状態↔表示の往復を明示的にテストしないと回帰に気づけない |
| E) ヒント/解説分割 | ヒントと解説（模範解答＋評価基準）が別々に生成され目次に出るか | 「課題→ヒント／解説」という仕様変更の検証 |
| F) XSS無害化 | プレビューに`<script>`を入力した際、`&lt;script&gt;`にエスケープされるか | ユーザー入力をinnerHTML的に描画する設計のため、最重要のセキュリティ項目 |
| G) 保存/一括保存/復元 | 単一タブ保存・複数キー一括保存・復元後に内容が戻るか | localStorage的な永続化はサイレント破損しやすく、復元まで含めて検証必須 |
| H) ひな形挿入 | 現在タブにひな形を挿入できるか | テンプレ機能の単体動作確認 |
| I) DL/ZIP | 各形式(md/txt/csv/json)のBlob生成、全部入りmd、ひな形のみDL、zip組み立て、**「全部入りにoverview混入なし」** | 結合ロジックのスコープ漏れ（余計なセクションが紛れ込む）を検出する唯一の項目 |
| J) ボタン自己チェック | 全`onclick`属性が実在する関数を指しているか機械的に走査 | 「押しても何も起きないボタン」をUIテストより安く確実に検出できる |
| K) セキュリティ | eval/Function不使用、CSVインジェクション対策、プレビューは先escape、外部送信はAPI1本のみ | 単一HTMLで配布するツールは監査されにくいため、自己チェックで担保するしかない |
| L) AI壁打ち（モック） | 全幅表示か、モック応答が返るか | 実APIを叩かずモックで結合だけ検証し、コストと不安定要素を排除 |

**カテゴリ別の生プロンプト例（Iの「overview混入」検証）：**

```text
カテゴリI「全部入りDL」の中に、以下の回帰テストを必ず含めてください。

- 本文編集セクション（受講生向け等）を一度書き換えてから、結合(buildCombined)を呼び出す
- buildCombinedの出力に overview セクションの内容が混入していないことを確認する
- このテストは「編集後の状態」で結合関数を呼ぶこと。生成直後の初期状態だけで
  テストすると、結合スコープのバグを見逃す可能性があるため必ず編集操作を挟むこと。
```

**なぜこの一文が要るか**：v7.0で実際に「1回目QAで赤が出たが、原因はテスト治具側の順序ミス（本文編集後に結合チェックすべきところを未編集のままチェックしていた）」という事例があった。テストは「実際の利用順序（編集→結合）」を再現しないと、本物のバグも治具のバグも両方見逃す。このプロンプトはその教訓をそのままルール化したもの。

## 4. プロンプト④：T3 実ファイル生成検証（jsdomを超える層）

```text
T3: pptx生成の実体検証を行ってください。
1. ツール内のpptx生成ロジック（台本→PptxGenJS呼び出し）を構文チェック
2. Node環境で実際に2種類のpptxファイル（受講生向け／講師向け）を生成
3. それぞれのファイルサイズ(bytes)を実測して出力
4. ファイルが0バイト、または極端に小さい場合は生成失敗として扱う
出力形式:
  ✅ <ファイル名>.pptx
      <バイト数> bytes  <ファイル名>.pptx
```

**なぜ**：jsdomはDOM操作のシミュレーションであり、`Blob`が「中身の伴った正しいOOXML/ZIP構造」を持つかまでは検証しない。バイト数という最も単純な指標でも、「0バイト＝壊れている」「異常に小さい＝中身が空」を機械的に検出でき、見開きUIのようなDOM層のテストでは絶対に拾えない不具合（同梱ライブラリの呼び出し漏れ、台本パースの失敗等）をここで初めて拾える。

## 5. プロンプト⑤：2回連続合格の強制

```text
T1〜T3を「1回目（最終確認）」として通しで実行し、結果をログに残してください。
その後、コードを一切変更しない状態で、もう一度「2回目（最終）」として
同じT1〜T3を実行してください。
両回とも全項目が✅の場合のみ「商用QA合格」と宣言してください。
片方でも結果が異なる場合（特にflakyな失敗）は、原因を特定するまで合格としないでください。
```

**なぜ**：1回しか実行しないテストは「たまたま通った」可能性を排除できない。特にDOM初期化やタイマー、非同期処理が絡むツールでは、実行順序やキャッシュ状態によって結果がブレるflaky testが起きうる。コード変更なしで2回連続合格を要求することで、再現性のない偶然の合格を出荷条件から排除する。1回目と2回目で結果に差分があれば、それ自体がテストの不安定性という重大な情報になる。

## 6. プロンプト⑥：失敗の切り分け（アプリ不具合 vs テスト治具の不備）

```text
テストで❌が出た場合、直接コードを「直す」前に、以下の手順で原因を切り分けてください。

1. 失敗したテストが検証している「期待される実際の操作手順」を言語化する
2. テストコード自身が、その操作手順を正しく再現できているか確認する
   （例：編集してから検証すべき箇所で、編集を省略していないか）
3. テストコードに不備がある場合 →「テスト治具の不備」として記録し、テストのみ修正する。
   アプリ本体のコードは変更しない。
4. テストコードは正しいのにアプリが期待通り動かない場合 →「アプリの不具合」として
   アプリ本体を修正し、修正差分を明示する。
5. どちらに分類したかを必ずQAログ・テストレポートの両方に明記する。
   「テストを直した」と「アプリを直した」を絶対に混同しない。
```

**なぜ**：この切り分けをしないと2種類の事故が起きる。①本当はアプリが正常なのに、テストに合わせてアプリ側を不要に書き換えて壊す。②本当はアプリが壊れているのに「テストがおかしいだけ」と誤魔化して出荷してしまう。v7.0では実際に①と②の境目にあたる事例（結合チェックの順序ミス）が発生し、原因を治具側と特定した上でテストのみ修正・アプリは無修正で押し通している。この記録があるからこそ「アプリは正常」という主張に説得力が出る。

## 7. プロンプト⑦：正直な制限事項の明記

```text
全テスト完了後、以下を必ずテストレポートに含めてください。

「正直な制限」セクションを作成し、以下を明記する：
- jsdom環境では検証できなかった項目（実ブラウザでしか動かないAPI、
  非同期処理の完走確認など）を具体的に列挙する
- 「設計上は動作するはずだが未実証」という表現を使い、
  検証済みであるかのような書き方をしない
- ヘッドレス実ブラウザ等、用意していない検証環境があれば明記する
```

**なぜ**：QAレポートの信頼性は「合格項目の多さ」ではなく「検証していない部分を隠さない誠実さ」で決まる。v7.0のレポートも「ブラウザ内の実PPT/実ZIPの非同期完走はjsdomで未実証」と明記して締めている。ここを曖昧にすると、後で実ブラウザで初めて顕在化する不具合が出たときに「テストは合格していたはずなのに」という信頼の崩壊につながる。

## 8. プロンプト⑧：最終テストレポート生成

```text
これまでのT1〜T3・2回分の結果をもとに、以下の構成でMarkdownのテストレポートを作成してください。

1. 総合判定（PASS/FAIL）
2. 1回目・2回目の結果を比較表で並べる（T1/T2/T3）
3. 今回の変更点と、それぞれに対応する検証結果の対応表
4. セキュリティ検証結果（XSSエスケープ、eval/Function不使用など）
5. 「テスト治具の不備」セクション（あれば。なければ明記しない）
6. 「正直な制限」セクション
7. 結論（バージョン番号・刻印日付・リリース可否）

トーンは簡潔・断定的すぎない・誇張なし。「合格」は実際に✅が並んだ項目にのみ使う。
```

**なぜ**：生ログ（QAログ）は実行証跡として残すが、人が読んで意思決定する単位として整形したレポートが別途必要。「変更点→検証結果」を対応表にすることで、依頼内容と検証内容のズレ（やったつもりでテストしていない機能）を視覚的に潰せる。

## 9. 実行順序まとめ（フロー）

```
①QA憲法を固定
   ↓
②T1構文チェック ──❌→ ここで停止、修正後①からやり直し
   ↓✅
③T2 jsdom（A〜L全カテゴリ）
   ↓
④T3 実ファイル生成（pptx等）
   ↓
　 ②〜④を「1回目」として実施
   ↓
　 コード変更なしで②〜④を「2回目」として再実施
   ↓
⑤失敗があれば「アプリ不具合／治具の不備」を切り分けて対処 → ②に戻る
   ↓（両回とも全✅）
⑥正直な制限を明記
   ↓
⑦最終テストレポート生成 → 合格・リリース可
```

## 付録：そのまま使える1メッセージ版（時短用）

毎回上記を全部書くのが面倒な場合の圧縮版。初回はこちらでも可（ただし切り分けロジックは必ず別途指示する）。

```text
このHTMLツールに対して商用QAを実施してください。
- T1: <script>部分をnode --checkで構文検証
- T2: jsdomでA)同梱ライブラリ B)生成/目次/見開き C)初期表示レイアウト
      D)編集→プレビュー反映 E)ヒント/解説分離 F)XSS無害化
      G)保存/一括保存/復元 H)ひな形挿入 I)DL各形式/全部入り（overview混入なきこと）/zip
      J)onclick実在チェック K)eval/Function不使用・CSVインジェクション対策・先escape
      L)AI壁打ちモック の各観点でテストし、passed/failed件数を集計
- T3: pptx生成ロジックを実際にNodeで走らせ、ファイルサイズを実測
これをコード変更なしで連続2回実行し、両方全✅で初めて「合格」と報告してください。
失敗時はアプリ不具合かテスト治具の不備かを必ず切り分け、嘘の合格報告はしないでください。
最後に未検証項目（実ブラウザ専用処理など）を「正直な制限」として明記してください。
```
